# Signals アーキテクチャ設計書：Signal Store（Bus）モデルへの移行

本ドキュメントは、Signals シンセサイザーエンジンの「段階1リファクタリング」によって導入された、共有データバス（Signal Store）を用いた新しいコアルーティングアーキテクチャの設計をまとめたものです。

## 1. 目的と概要
従来の `Signals` では、モジュールの出力オブジェクトを直接次のモジュールの `process([inputs])` メソッドに渡す「直接接続」のグラフ構造を採用していました。今回のリファクタリングにより、モジュール間の依存関係を分離し、すべての信号データを一元管理する **Signal Store（共有ワークスペース）** モデルへと移行しました。

これにより以下のメリットがもたらされます：
- **モジュール間の完全な疎結合化:** 任意の信号が単一のキー名を通じてルーティングされるため、1つのLFOを複数箇所へアサインするなどの複雑なルーティングが極めて容易になります。
- **実行ループのフラット化:** 再帰的な呼び出しがなくなり、$O(1)$ベースの平坦なループによって各ノードが独立して処理されるため、安定性と拡張性が向上します。
- **CLIとGUIへのシームレスな統合:** ノードエディタ上の「ワイヤー」メタファーと1対1で対応するキー名管理が可能になり、CLI単体での即席パイプライン構築も容易になります。

---

## 2. Core レイヤーの責務と関係性

新しいアーキテクチャは主に3つのコアコンポーネントで構成されます。

### 2.1. `SignalStore`
- **ファイル:** `src/signals/core/store.py`
- **責務:** グラフ全体で共有される辞書型のデータストア。バスキー（例: `bus_osc1_0`, `wire_99`）をキーとして、`Signal` オブジェクトを格納・提供します。
- **機能:** `get(key, default)` による信号の読み出しと、`set(key, signal)` による信号の書き込みを提供します。要求されたキーが存在しない場合は、デフォルトとして無音の `AUDIO` 信号（0.0）を返却することで、未接続状態のフェイルセーフを担います。

### 2.2. `StoreNode`
- **ファイル:** `src/signals/core/store_node.py`
- **責務:** 従来のドメインロジックを持つ `Module` インスタンスをラップし、`SignalStore` アーキテクチャに適合させるアダプタ。
- **機能:**
  - `input_keys: Dict[int, str]` と `output_keys: Dict[int, str]` を保持します。
  - `process(store: SignalStore)` メソッドが呼ばれると、`input_keys` を元に `store` から必要な入力信号を収集し、内部の `Module.process(inputs)` を実行します。その後、戻り値の信号群を `output_keys` に従って `store` へ格納します。

### 2.3. `ModuleGraph`
- **ファイル:** `src/signals/processing/graph.py`
- **責務:** パッチ情報からノードを構築し、依存順序（Kahnのアルゴリズム）に従って処理を進行させるグラフマネージャ。
- **変更点:** 各ノードオブジェクトは `StoreNode` を包含します。`process_sample()` の実行時は、依存順序でソートされた配列を回しながら `node.store_node.process(self.store)` をフラットに呼び出すだけのシンプルなループ構造になりました。

## 3. 既存 Module API とのインターフェース
既存のモジュール（Oscillator, Filter, VCAなど）のロジックやアルゴリズムは **一切変更していません**。`StoreNode` が「コンポジション（包含）」の形でこれをラップしています。

**処理の流れ:**
1. `StoreNode` は、自身が持つ `input_keys` に定義されたバスキー（例: `{0: "bus_audio", 1: "bus_cutoff"}`）を使って `SignalStore` を検索し、入力シグナルのリストを構築します。
2. そのリストを `self.module.process(inputs)` に渡します（従来通りの呼び出し）。
3. `Module` は純粋にドメインロジック（波形の生成、フィルタリングなど）を実行し、出力シグナルのリストを返します。
4. `StoreNode` は返された出力を、自身の `output_keys`（例: `{0: "bus_filter_out"}`）に従って `SignalStore` に `set()` します。

これにより、既存の豊富なモジュール群のテスト資産や安定性をそのまま維持しつつ、アーキテクチャの刷新を実現しています。

## 4. YAML (Patch) の互換性確保のための内部マッピング戦略
既存の `Patch` YAML フォーマットは `connections:` リストを用いて「どこの出力ポート（from）」から「どこの入力ポート（to）」へ繋ぐかを定義しています。この互換性を完全に保つため、`ModuleGraph._build_graph()` 時に暗黙的なバスキーの自動生成とマッピングを行っています。

**内部マッピングの手順:**
1. YAML から `connections`（例: `from: osc1.0`, `to: filter1.0`）を読み取ります。
2. これらを直接のオブジェクト参照とする代わりに、システム内部で一意のバスキー（例: `wire_osc1_0`）を自動生成します。
3. 送信元ノードの `StoreNode.output_keys[0]` に `wire_osc1_0` を登録します。
4. 受信先ノードの `StoreNode.input_keys[0]` にも同じ `wire_osc1_0` を登録します。
5. （フェイルセーフとして、接続先がない出力ポートに対しても自動的に `wire_{module_id}_{output_idx}` というキーを割り当て、全体の出力結果として収集可能にしています）

この「仮想ワイヤー」方式により、将来的にはビジュアルノードエディタ上で線を繋ぐ際、その線に振られたID（例: `wire_99`）をそのまま JSON で送るだけでバックエンドが解釈できる、完全な統合が達成されます。

## 5. CLI レイヤーと Core の関係
Signals の機能を「ちょっと波形を作りたい」「この音声にローパスだけかけたい」といったUNIX的な単体ツールとして呼び出せるよう、薄いラッパーとなるCLIレイヤーを追加しました。

- **ファイル:** `src/signals/cli.py` および `pyproject.toml`
- **エントリーポイント:** `signals-osc`, `signals-filter` 等
- **構成方法:**
  1. CLI スクリプトはローカルに `SignalStore` を単独でインスタンス化します。
  2. `StoreNode` を用いて、対象モジュールと入出力（`OutputWav`）をセットアップします。
  3. `argparse` で受け取ったパラメータをモジュールに設定し、`store` 上の固定キー（例: `bus_audio`、`bus_in`、`bus_out`）を介してデータをやり取りする「即席パイプライン」を構築します。
  4. ファイル入力がある場合（Filterなど）は、標準ライブラリの `wave` でWAVファイルを読み込み、それを `store` に流し込みながらサンプル単位で処理を回します。

このアプローチにより、大掛かりな YAML パッチや `SynthEngine` を介さずとも、コア層の `SignalStore` アーキテクチャをそのまま使い回すことができ、ロジックの重複が排除されています。
