# RNBO C++ エクスポートコードの統合に関する検討レポート

本レポートは、Cycling '74 Max の RNBO を用いて構築・エクスポートされた C++ コードを Signals フレームワークに統合する計画について、現状のアーキテクチャ設計での実現可能性、必要な更新、および今後の Rust への移植が与える影響について検討したものです。

## 1. 現段階の設計での統合の実現可能性

**結論：現段階の設計でも統合は可能ですが、大きなアーキテクチャの更新とラッパー層の実装が不可欠です。**

現在の Signals は Python で実装されており、各モジュール（`Oscillator`, `Filter`, `Envelope`など）は `src/signals/core/module.py` に定義された `Module` 抽象基底クラスを継承しています。
現状、`process()` メソッドはサンプル単位 (1サンプルごとの処理) で呼ばれる設計 (`list[Signal]` を返す) になっています。

RNBO からエクスポートされた C++ コードを Python に統合すること自体は、Pybind11 や CFFI、Cython 等を用いて C++ のバインディングを作成することで技術的に十分に可能です。RNBO のエクスポートコードは標準的な C++ クラスとして生成されるため、それを Signals の `Module` クラスを継承した Python ラッパークラスに内包させることで、グラフやパッチのルーティング機構をそのまま活かすことができます。

## 2. 必要な更新内容

C++ (RNBO) コードを統合し、実用的なパフォーマンスとアーキテクチャの一貫性を保つためには、以下の更新が必要です。

### a) C++ バインディング層の追加
RNBO が出力する C++ の `RNBO::CoreObject` を Python から利用するために、Pybind11 などを利用した拡張モジュール (C++ ラッパー) をビルド・連携する仕組みをビルドパイプライン (現在は `hatchling` を使用) に追加する必要があります。

### b) ブロック処理（バッファベース処理）への移行
現在、Signals の `Oscillator` などのモジュールは `process()` の中で1サンプルごとに処理を行う設計となっています (サンプルごとのループ処理)。
RNBO を含む多くの C++ 音響処理フレームワークは、オーディオバッファ（例: 64〜1024サンプル）を一括で処理するブロックベースの設計が基本であり、パフォーマンス上もそれが前提です。
Python と C++ 間の呼び出しオーバーヘッド（GILの取得・解放や型変換）を最小限にするため、1サンプルごとにC++関数を呼ぶのではなく、Numpy 配列などのバッファを C++ に渡し、ブロック単位で処理を完了させるように、`Module` インターフェースおよび `ModuleGraph` の処理ループ (`process_sample`) を根本的にブロック処理対応へ移行させる必要があります。

### c) パラメータ更新とイベント処理の同期
RNBO オブジェクトのパラメータ設定や MIDI イベント等のトリガーは、RNBO のイベントモデルに準拠しています。Signals の現在の `set_parameter` メソッドおよび `SequenceEvent` を、RNBO のパラメータインデックスおよびイベントスケジューラに適切にマッピング・変換するブリッジ機能の実装が必要です。

## 3. 今後の Rust への移植による影響

**結論：Rust への移植は、RNBO (C++) との統合において非常に好相性であり、むしろPythonに直接統合するよりもパフォーマンスと安全性の面で有利に働きます。**

### メリットとシナジー
- **ゼロコストバインディング:** Rust と C++ の相互連携ツール（`cxx` クレートや `bindgen` クレート）は非常に成熟しています。Python-C++ 間のような重い呼び出しオーバーヘッドを気にすることなく、Rust エンジンから直接 RNBO の C++ コードを高速に呼び出すことができます。
- **メモリ安全性とパフォーマンス:** Rust の所有権モデルと C++ のパフォーマンスが融合することで、オーディオエンジンの要件である「低レイテンシ・リアルタイム処理」を極めて高いレベルで実現できます。
- **ブロック処理の自然な実装:** Rust でオーディオエンジンを作り直す際、CPAL などの標準的なオーディオクレートはブロックベース（バッファベース）のコールバック駆動です。したがって、前述の「ブロック処理への移行」は Rust 移植の際に自然と設計に組み込まれることになります。

### 今後の移行ステップに向けた推奨事項
もし Rust への移行が近いうちに予定されているのであれば、現在の Python アーキテクチャで C++ 統合のオーバーヘッド解決（ブロック処理化など）に大きな開発コストをかけるよりも、以下のステップを踏むことを推奨します。

1. **フェーズ1 (現行 Python)**: 概念実証 (PoC) として、バッファサイズの概念を取り入れつつ Pybind11 等で最小限の RNBO ラッパーモジュールを作成し、グラフ処理上のルーティングテストを行う。
2. **フェーズ2 (Rust 移植)**: コアエンジン (Graph, SignalStore, Scheduler) を Rust で再構築する際、初期段階から C++ (RNBO) モジュールのネイティブホスティングを前提としたアーキテクチャ（`cxx` を用いたブリッジングとブロックベースのバス処理）を設計する。

これにより、無駄な Python 拡張の開発コストを抑えつつ、将来のモダンで高速な Rust + RNBO C++ プラットフォームへとスムーズに移行できます。
